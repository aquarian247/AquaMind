## Executive Summary

The analysis of both PDF reports reveals **explicit transfer mapping information** that can be used to link freshwater containers to sea rings for the Fishtalk→Aquamind migration. The freshwater report (page 15) contains a detailed "ÚTSETINGARÆTLAN" (Transfer Out Plan) table that maps specific freshwater stations and halls to sea destinations. The sea report (pages 29-47) provides detailed ring-level data for each sea area, including deployment dates and batch identifiers.

### Key Findings

1. **Freshwater report page 15** contains explicit FW→Sea transfer plans with:
   - Source FW station + hall (e.g., "S03 Norðtoftir (18B)")
   - Destination sea location (e.g., "Hvannasund S", "A15 Tvøroyri")
   - Fish counts being transferred
   - Average weight at transfer
   - Transfer week/date

2. **Sea report pages 29-47** contain detailed ring-level data for each sea area with:
   - Ring numbers
   - Deployment dates
   - Batch identifiers (encoding supplier, year class, FW input date)
   - Fish counts per ring
   - Performance metrics

3. **Batch identifiers** in sea report encode critical information:
   - **SF** = Stofnfiskur, **BM** = Benchmark (supplier)
   - **25/26** = year class
   - **(JUN 24)**, **(AUG 24)**, **(SEP 24)** = FW input/start dates
   - **[*MAD]** = special designation

## Mapping Strategy for Migration

### Approach 1: Direct Transfer Plan Mapping (Most Reliable)

Use the **Freshwater Report Page 15** transfer plan as the primary source:

1. **Extract transfer records** from the "ÚTSETINGARÆTLAN" table
2. **Map each FW station + hall** to destination sea area
3. **Use fish counts and transfer dates** to validate against sea report ring data
4. **Create FW→Sea linkage** in Aquamind based on:
   - Source: `infrastructure_freshwaterstation` + `infrastructure_hall`
   - Destination: `infrastructure_area` (sea location)
   - Transfer date
   - Fish count
   - Batch identifier (from sea report batch codes)

### Approach 2: Batch Identifier Matching (Supplementary)

Use the **batch identifier codes** from sea report to trace back to FW origins:

1. **Extract batch codes** from sea report ring data (e.g., "SF [BM] 25 (JUN 24)")
2. **Parse components**:
   - Supplier (SF = Stofnfiskur, BM = Benchmark)
   - Year class (25, 26)
   - FW input date (JUN 24, AUG 24, SEP 24)
3. **Match against FW report batch names** (e.g., "BM Jun 24", "Stofnfiskur Okt 25")
4. **Cross-reference with Ext_Populations_v2.PopulationName** in Fishtalk database

### Approach 3: Fish Count Reconciliation (Validation)

1. **Sum fish counts** from FW transfer plan for each destination
2. **Compare with total fish counts** in corresponding sea area
3. **Flag discrepancies** for manual review
4. **Account for mortality** between transfer and reporting dates

## Data Quality Observations

### Strengths

1. **Explicit transfer plans** in freshwater report provide clear FW→Sea mappings
2. **Hall-level granularity** allows precise container-to-ring linking
3. **Batch identifiers** in sea report encode supplier and timing information
4. **Fish counts** enable validation through reconciliation

### Challenges

1. **Date discrepancies**: Sea report shows October 2024 deployments, while FW report shows December 2025 transfers (may be different cohorts)
2. **Fish count mismatches**: Planned transfers (2.8M) vs. actual sea area counts (1.4M) suggest multiple transfer events or mortality
3. **Missing areas**: A15 Tvøroyri not found in reviewed sea report pages (may be in pages 44-47)
4. **Batch code variations**: Multiple encoding schemes (SF [STØFN], SF [BM], SF [OKT]) require careful parsing

## Recommendations for Migration

### Immediate Actions

1. **Extract complete transfer plan** from Freshwater Report page 15
2. **Review sea report pages 44-47** to find A15 Tvøroyri and other missing areas
3. **Query Fishtalk database** for:
   - `Ext_Populations_v2.PopulationName` matching batch identifiers
   - `SubTransfers` records for FW→Sea movements
   - `Ext_Inputs_v2` for egg input tracking

### Migration Implementation

1. **Create ExternalIdMap entries** for each transfer event:
   - `source_model = "FWTransfer"`
   - `source_identifier = "{FW_Station}_{Hall}_{Sea_Area}_{TransferDate}"`
   - Store metadata: fish_count, avg_weight, batch_identifier

2. **Generate BatchTransferWorkflow records** for each FW→Sea transfer:
   - Link source containers (FW halls) to destination containers (sea rings)
   - Set transfer dates from FW report
   - Populate fish counts and biomass

3. **Validate transfers** using:
   - Fish count reconciliation (FW plan vs. sea area totals)
   - Date consistency checks
   - Batch identifier matching

### Data Gaps to Address

1. **Missing transfer dates**: Some FW→Sea transfers lack specific dates (only week numbers)
2. **Ring-level assignments**: Need to determine which specific rings received fish from which FW halls
3. **Mortality accounting**: Reconcile fish count differences between FW transfer and sea arrival
4. **Historical transfers**: Reports show recent data; need to identify older FW→Sea transfers for complete migration

## Conclusion

The PDF reports provide **sufficient information to establish FW→Sea transfer mappings** for recent cohorts (2024-2026). The freshwater report's transfer plan (page 15) is the most reliable source for explicit FW station + hall → sea area mappings. The sea report's ring-level data provides validation through fish counts and batch identifiers.

For a complete migration, this approach should be:
1. **Extended to historical data** by querying Fishtalk database tables (SubTransfers, Ext_Populations_v2)
2. **Validated against** Ext_Inputs_v2 batch keys (InputName + InputNumber + YearClass)
3. **Supplemented with** heuristic matching for transfers not explicitly documented in reports

The combination of **explicit transfer plans**, **batch identifiers**, and **fish count reconciliation** provides a robust framework for establishing non-ambiguous FW→Sea links in the Aquamind migration.
